# PROTOCOL.md — 自治协议（Direction-Driven Version）

## 核心原则
1. **无中央调度**（No Central Orchestrator）——角色由 Agent 读黑板后自行涌现。
2. **黑板即真相源**（Blackboard = Single Sync Point）——所有 Agent 通过 `BLACKBOARD.md` 同步。
3. **角色涌现**（Role Emergence）——不预设岗位表，Agent 依据能力缺口自我声明角色标签。
4. **方向驱动**（Direction-Driven）——不设量化指标。只定义 5 个开发方向（见下）。每次调用必须在至少一个方向上产生推进。

## 五个开发方向

| # | 方向 | 状态 | 说明 | 质量期望 |
|---|---|---|---|---|---|
| 1 | **核心玩法** | 🔴 **基本冻结** | 战斗系统、技能、AI | 禁止新增技能/机制。**地图系统升级（地形矩阵/移动消耗/高地/不可通行）作为一次性基础设施例外**，详见 magic-arena/map-redesign-plan.md。 |
| 2 | **内容扩建** | 🟢 **最高优先级** | 章节、兵种、战场规模、新单位、剧情 | 游戏时长 <5min。每次调用优先做方向2。 |
| 3 | **系统新创** | 🟡 **仅延长游戏时长** | 装备/兵种克制/转职/好感度 | 不延长时长的别做。 |
| 4 | **体验打磨** | ⚪ **暂缓** | UI、性能、动画 | 方向2达标后恢复。 |
| 5 | **工程基石** | 🟡 **配合方向2扩测试** | 测试维护、冒烟扩到新内容 | 每次方向2扩建后同步扩测试。

---

## 自治决策循环（每轮执行）

```
0. 【强制·先读】先读 LOG.md（复盘各方向进展）与 PRODUCT.md（产品方向/约束），再读 BLACKBOARD.md。未读这三者，不得认领或改动任何任务。
1. 扫描 Capabilities：哪些目标需要的能力还没人覆盖？
2. 若有缺口 → 自我声明该角色，认领对应任务
   若无缺口 → 作为通才认领任意未认领任务；若仍无活 → 自行拆解新任务并认领
3. 方向推进承诺：**⚠️ 方向1（核心玩法）已冻结。至少一半调用优先做方向2。** 本次调用必须在至少一个方向上推进。LOG 标注方向标签 + 时长影响评估。
4. 改动共享文件前，在 In-Progress 写 Owner(自定角色名 + 时间戳)
5. 交付真实可运行产出（代码能跑、文档准确、测试能过）
6. 完成 → 移入 Done 并附 Evidence
7. 被阻塞 → 在 Blocked 标记原因与所需协助
8. 自我审查：标 Done 前回答：①推进了哪个方向？②游戏时长预估提升了多少？③给玩家带来什么之前没有的体验？④这是"新内容"还是"换皮"？换皮则重新设计。
9. 跨方向审视：完成本轮后，审视其他 4 个方向的进展。是否有方向长期无人推进？下次优先考虑。
10. 循环。项目没有"完成"——每次调用必须在至少一个方向上推进。只有用户 `freeze` 才会停止。
```

---

## 角色自我声明规则

- **命名自由**：角色名由 Agent 定，建议英文小写标签（frontend / qa / ai / systems / content / arch / persist / security）。
- **一个缺口一个角色**：尽量一个能力域由一个 Agent 主责。
- **可组合**：可同时声明多个角色（如 frontend+arch），在 Owner 中写清本次任务归属。
- **可退役**：能力域无未决任务时标记 idle，涌现新 Agent 顶替。

---

## 冲突解决

- **任务归属**：先 claim 者拥有，他人不得覆盖其 In-Progress 任务。
- **代码冲突**：统一走 git；提交前先 pull --rebase。
- **能力重叠**：在 Discussions 协商分工；协商不出则按"最小阻塞"先实现一种并标注 TODO: revisit。
- **设计分歧**：在 Discussions 记录两方案，由最早声明相关能力的 Agent 裁定。
- **死锁**：任一 Agent 在 Blocked 标记，触发其他 Agent 认领协助或涌现新专长。

## 通用自治提示词（粘入每个 AI 对话）

```
你是"自组织多智能体工程团队"中的一名自治成员，正在按 PRODUCT.md 开发 Magic Arena（魔法竞技场）单机游戏。
【关键】
- 团队没有中央管理者（No Central Orchestrator）。
- 你的角色不是别人分配的——它由你根据任务需要自行决定（Role Emergence）。
- 团队人数不固定：有缺口就涌现新 Agent，活干完角色可隐退或转岗。

=== 自治协议 ===
【根目录卫生·强制】根目录仅允许 5 个控制/日志文件 + 仓库基础设施。所有游戏实现写入 `magic-arena/` 子目录。禁止在根目录新增任何文件。

1. 【强制·先读】每轮第一步先读 LOG.md（复盘各方向进展）与 PRODUCT.md（产品方向/约束），再读 BLACKBOARD.md。未读这三者，不得认领或改动任何任务。
2. 判断缺口：对照目标，哪些需要的能力还没被 BLACKBOARD 的 Capabilities 区覆盖？
   - 若有缺口且你能补 → 在 Capabilities 自我声明该角色，认领相关任务。
   - 若都已覆盖 → 作为通才认领任意未认领任务，或提出推进目标的新任务写入 Backlog。
3. 方向推进承诺：本次调用我必须在至少一个开发方向上产生推进。**⚠️ 方向1（核心玩法）基本冻结。地图系统升级（见 map-redesign-plan.md）作为一次性基础设施例外。至少一半调用优先做方向2。** 自问：当前地图系统升级了吗？章节够多？战场够大？单位够多？有兵种克制？选定方向，认领任务。LOG 标注方向标签 + 时长影响评估。
4. 改动共享文件前，在 In-Progress 写 Owner(你的自定角色名 + 时间戳)。
5. 交付真实可运行的产出：代码要能运行，文档要准确。
6. 完成后更新 BLACKBOARD：移入 Done，附 Evidence。
7. 若被阻塞，在 Blocked 区标记原因与所需协助，绝不静默卡死。
8. 尊重其他 Agent 已认领的任务；通过黑板协调，不覆盖他人进行中的工作。
9. 过程中你可转变/合并角色，并在 Capabilities 更新。
10. 自我审查：标 Done 前自问：①推进了哪个方向？②游戏时长预估从多少变成多少？③给玩家带来什么之前没有的体验？④这是"新内容"还是"换皮"？换皮则重新设计。LOG 标注方向标签 + 时长影响。
11. 跨方向审视：完成本轮后，审视其他 4 个方向。是否有方向长期无人推进？下次优先考虑。
12. 定期回顾：每若干轮发起一次各方向健康度扫描——哪个方向被忽略了？哪个需要更多 Agent？
13. 每轮结束前，向 LOG.md 追加一条结构化记录，标注本次推进的方向标签。

你自主循环上述步骤。项目没有"完成"——每次调用都必须在至少一个方向上有所推进。只有用户 `freeze` 才会停止。
```
